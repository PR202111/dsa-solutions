# Contributing Guidelines

We welcome contributions to this repository! Whether you're fixing bugs, improving documentation, or optimizing solutions, here's how to help.

## Ways to Contribute

1. **Bug Fixes**: Report and fix any issues in the solutions
2. **Optimizations**: Suggest or implement more efficient algorithms
3. **Documentation**: Improve comments and explanations in code
4. **New Solutions**: Add alternative approaches to existing problems
5. **Test Cases**: Add comprehensive test cases for each solution

## Getting Started

1. Fork the repository
2. Create a feature branch (`git checkout -b feature/improvement`)
3. Make your changes
4. Test your code thoroughly
5. Commit with clear messages (`git commit -m 'Add improved solution for problem X'`)
6. Push to your branch (`git push origin feature/improvement`)
7. Open a Pull Request

## Code Style Guidelines

- Use clear, descriptive variable names
- Add comments explaining complex logic
- Follow C conventions and best practices
- Include proper error handling
- Test edge cases

## Naming Conventions

- Functions: `descriptive_function_name()`
- Variables: `variable_name` (snake_case)
- Constants: `CONSTANT_NAME` (UPPER_SNAKE_CASE)
- Files: `meaningful_name.c`

## Pull Request Process

1. Update README.md if adding new solutions
2. Provide description of changes
3. Reference any issues addressed
4. Ensure code compiles without warnings
5. Add test cases if applicable

## Issue Reporting

When reporting issues:
- Describe the problem clearly
- Include compilation/runtime errors
- Suggest possible solutions if applicable
- Provide environment details (OS, compiler version)

## Code Quality

- No hardcoded values; use constants
- Functions should do one thing well
- Maximum function length: 50-100 lines
- Use meaningful variable names, not single letters (except loops)

## Documentation

When adding new solutions:
1. Add brief problem description
2. Include algorithm explanation
3. Document time and space complexity
4. Add example usage in comments
